# 解决方案
## 目标
- 降低 dashboard 首屏接口响应体大小，优先保证“列表信息先到、头像后到”。
- 在不影响现有功能（新增/编辑宝宝、查看详情、本地模式）的前提下优化加载速度。

## 根因分析
- 当前列表加载路径会直接查询 `babies` 全字段（`select('*')`），包含 `avatar`。
- `avatar` 目前是 base64 Data URL，体积通常比二进制图像更大（约增加 33%），且会随着宝宝数量线性放大响应体。
- 列表页只需要名字、生日、性别等轻量字段；头像属于非关键渲染内容，不应阻塞首屏。

## 推荐方案（分两阶段）
### 阶段1：低风险快速优化（先做）
1. 列表查询去掉 `avatar`
- `refreshBabies/loadBabiesFromSupabase` 改为仅查询列表必要字段：`id,name,birth_date,gender,premature_birth_date`。
- 列表先渲染默认头像（性别 emoji 或占位图），不等待真实头像。

2. 头像按需加载（懒加载）
- 新增轻量接口方法：按 `babyId` 单独读取 `avatar`。
- 在卡片进入视口后再请求头像（IntersectionObserver），或在卡片渲染后异步批量并发限制加载（如并发 3~4）。
- 请求成功后仅更新对应卡片头像，失败时保持占位图。

3. 增加前端缓存
- 内存缓存：`Map<babyId, avatar>`，同一会话避免重复拉取。
- 可选 localStorage 缓存（带版本/时间戳），减少重复访问。

预期收益：
- 首屏接口体积显著下降（常见可下降 60%~95%，取决于头像数量与大小）。
- 列表可更快可交互，头像延后渐进显示。

### 阶段2：结构性优化（建议后续）
1. 头像迁移到对象存储（Supabase Storage）
- 数据库不再存 base64，改存 `avatar_path` 或 `avatar_url`。
- 上传时在前端压缩并上传文件（例如 WebP/JPEG，限制边长和大小），数据库仅保存路径。

2. 列表使用缩略图
- 存两种尺寸：`thumb`（列表）+ `full`（详情）。
- 列表仅加载 `thumb`，详情页再加载 `full`。

3. 清理历史 base64 数据
- 提供一次性迁移脚本：读取旧 `avatar(base64)` -> 上传到 Storage -> 回写 `avatar_path` -> 清空旧字段。

预期收益：
- 长期控制数据库与传输成本。
- 支持更多用户和更多宝宝数据时仍保持稳定性能。

## 兼容与风险控制
- 本地模式（localStorage）保持原行为，优化仅针对 Supabase 路径。
- 通过特性开关控制：`ENABLE_AVATAR_LAZY_LOAD=true`，可快速回滚。
- 头像加载失败不影响主流程，仅显示占位图。

## 验收标准
- 列表 API 首次响应体积较当前下降 >= 50%。
- 在 10 个宝宝、每个头像约 100KB base64 的场景下，首屏可交互时间明显缩短（建议目标 >= 30%）。
- 功能回归通过：新增/编辑宝宝、复制 ID、进入详情页、导入宝宝流程均正常。

## 实施顺序建议
1. 先落地阶段1（1~2 个迭代内可完成）。
2. 线上观察性能与错误率（1 周）。
3. 再推进阶段2（存储迁移与历史数据清理）。
